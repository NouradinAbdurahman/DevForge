# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A production-grade cross-platform development workstation lifecycle manager ("dotfiles" repo and then some), now a mature local-first developer platform. `bootstrap.sh` + `scripts/` (Layer 1, pure bash, must run correctly on a brand-new machine before any of its own tooling exists) provisions, backs up, updates, diagnoses, inventories, and releases a real workstation, and ships copyable project templates. On top of that, `cli/` (Layer 2, Node.js, installed by bootstrap itself) is the Core CLI framework; `registry/` (Layer 4) and `plugins/` (Layer 3) are manifest-driven extension points it reads. See `docs/PlatformArchitecture.md` for the full four-layer design and the multi-release roadmap — everything in this file's Layer-1 sections below is unaffected by it. Deep-dive docs live in `docs/` (Architecture, PlatformArchitecture, ArchitectureDiagrams, CLI, CommandReference, KeyboardShortcuts, MigrationGuide, TUI, WorkspaceManager, Scripts, GitHubActions, Security, ReleaseProcess, Preferences, Profiles, Recipes, ProjectGenerator, PluginSdk, AIAssistant, EnvironmentGraph, BenchmarkGuide, RepairGuide, CompatibilityEngine, Inventory, Templates, Troubleshooting, Customization, CONTRIBUTING); the README links to all of them.

## Commands

Every command below also works as `./devforgekit <name>` (`./devforgekit doctor` ==
`./scripts/doctor.sh`) - `devforgekit` (repo root, no extension) is a pure
dispatcher, see the Architecture section.

```bash
./bootstrap.sh                 # full provision: Homebrew, mise, dotfiles, editors, services
./bootstrap.sh --dry-run --yes # validate everything with no side effects (what CI runs)
./bootstrap.sh --profile flutter  # or --minimal/--full/--profile <name> - install a Brewfile subset
./scripts/validate.sh          # bash -n + ShellCheck + Brewfile (+ profiles) + mise.toml + JSON + YAML + Markdown
./scripts/check.sh             # fast PASS/WARNING/FAIL sweep + health score
./scripts/doctor.sh [--fix]    # deep diagnostics + PATH manager + health score; --fix repairs missing PATH entries
./scripts/backup.sh            # capture live config into the repo, commit+push if changed
./scripts/restore.sh           # dotfiles/editors only, no packages/services
./scripts/update.sh            # upgrade every managed toolchain, restart services
./scripts/services.sh <start|stop|restart|status>
./scripts/cleanup.sh           # reclaim disk space across every cache
./scripts/report.sh            # writes reports/system-report.txt
./scripts/inventory.sh         # writes reports/{system,hardware,software,brew,fonts,extensions,services,databases,network}.md
./scripts/preferences.sh <backup|restore|status>  # macOS UI preferences (Dock, Finder, etc.)
./scripts/profile.sh <list|show|use> [name]       # manage install profiles (profiles/<name>/Brewfile)
./scripts/release.sh <patch|minor|major>          # bump VERSION, draft CHANGELOG, commit, tag, push
```

No single script installs shellcheck/mise/yq for you when missing — `validate.sh` and others degrade gracefully (log a warning, skip that check) rather than failing when an optional tool isn't present.

### Node Core CLI (`cli/`)

```bash
cd cli && npm install         # install dependencies (bootstrap.sh does this automatically via ensure_cli_dependencies)
npm run lint --prefix cli     # eslint
npm test --prefix cli         # node --test test/
node cli/bin/devforgekit.js --help   # run it directly without going through the root dispatcher
```

Once `cli/node_modules` exists, `./devforgekit <any command except install/bootstrap>` transparently delegates here instead of the bash scripts below — see the Architecture section.

## Critical constraint: bash 3.2 (Layer 1 only)

This applies to `bootstrap.sh`/`scripts/*.sh`/`common.sh`/`colors.sh` — **not** to `cli/` (Node, ESM, no bash-version constraint). Every Layer-1 script must run correctly under the stock macOS `/bin/bash` (3.2.57) — a fresh Mac has nothing newer, and `bootstrap.sh` has to bootstrap Homebrew itself. **Never use bash 4+ features**: no `declare -A` (associative arrays), no `declare -g`, no `mapfile`/`readarray`, no `${var,,}`/`${var^^}`. `scripts/doctor.sh`'s PATH-duplicate check deliberately uses `sort | uniq -d` instead of an associative array for this reason — follow that pattern if you need similar bookkeeping. Validate any new script with `/bin/bash script.sh`, not just whatever `bash` resolves to on PATH (this repo's own `.zshrc` puts Homebrew/GNU coreutils first, which can silently hide bash-3-incompatible code during interactive testing).

Similarly, prefer flags that behave identically under BSD and GNU userland tools, since this machine's PATH may shadow BSD tools with Homebrew's GNU coreutils (`gnu-sed`, `grep`, `findutils`, etc. — see `.zshrc`). Example: disk-space checks use `df -Pk` (POSIX single-line, 1024-byte blocks) rather than BSD-only `df -g`, because `df -g` errors under GNU coreutils' `df`.

## Critical constraint: `set -e` / `pipefail` hazards

Every script runs under `set -Eeuo pipefail`. Two failure modes have already bitten this repo in CI — watch for both when adding code:

1. **Never call `print_summary` as a bare statement.** Its last command returns non-zero whenever any step actually `FAIL`ed, so `print_summary; STATUS=$?` under `set -e` aborts the script *at that line*, before `STATUS` is even assigned — skipping everything after it (timing output, in `bootstrap.sh`'s case the whole final tool checklist). Always wrap it: `if print_summary; then STATUS=0; else STATUS=1; fi` (or `if print_summary; then exit 0; else exit 1; fi` if nothing else follows). Same logic applies to any other function whose own exit status is meaningful (see `bootstrap.sh`'s `_check_tool`, which had the identical bug with a bare `command_exists "$2"`).
2. **A bare pipeline where an early/middle stage can return non-zero** (e.g. `find` on a missing directory, `grep` finding no matches) aborts the script under `pipefail`, even though the last stage in the pipe succeeds — `pipefail` takes the rightmost *non-zero* exit code, not just the last command's. Guard with `|| true` when "found nothing" is a valid outcome (see `scripts/inventory.sh`'s `find`/`grep` calls).

Prefer an `EXIT` trap over an `ERR` trap for "something went wrong" banners: with `errtrace` (`-E`) enabled, an `ERR` trap still fires inside `run_step`/`run_step_optional` even though they run their command under a local `set +e` — trap-on-ERR firing is gated by *command position* (if/while/&&/||), not by whether errexit is currently on. `bootstrap.sh` uses an `EXIT` trap instead, which only fires once, when the script is truly terminating.

## Architecture

- **`scripts/colors.sh`** — ANSI color/symbol constants, disabled automatically when stdout isn't a TTY.
- **`scripts/common.sh`** — the shared library every other script sources. Provides: logging (`log_info`/`log_success`/`log_warn`/`log_error`/`log_section`/`log_step`), timers, OS/arch detection (`os_is_macos`, `os_arch`, `os_brew_prefix` — handles Apple Silicon `/opt/homebrew` vs Intel `/usr/local`), `net_has_internet`, `confirm` (honors `DEV_SETUP_ASSUME_YES=1` and non-interactive shells), `version_of` (first line of `<bin> --version`-style output, or "not installed"), the idempotent `fs_safe_copy` (skips identical files, backs up differing ones as `<file>.backup-<timestamp>` before overwriting), `config_file_pairs`/`preference_domain_pairs` (single source of truth for which repo file maps to which `$HOME` path, and which `defaults` domain maps to which preference backup file), `restore_zsh`/`restore_git`/`restore_mise`/`restore_editor`/`backup_editor`, the `SERVICE_LIST` array and `service_start_all`/`stop_all`/`restart_all`/`status_all`/`verify_all`, and the fault-tolerant step runner (`run_step`, `run_step_optional`, `record_result`, `print_summary`) that every script uses to report PASS/WARNING/FAIL without one failed step killing the whole run.
- **`bootstrap.sh`** — sources `common.sh` and orchestrates, in order: preflight (macOS/arch/internet/disk checks) → Homebrew (`ensure_homebrew` + `brew bundle`) → runtimes/config (`restore_mise`/`restore_zsh`/`restore_git`/`restore_editor` for both editors) → services (start + verify) → `scripts/report.sh` → colored summary with execution time. Supports `--dry-run` (validates without installing/copying/starting anything — this is what CI runs), `--skip-services`, `-y/--yes`.
- **`scripts/*.sh`** are thin, single-purpose entry points built on the same `common.sh` functions — `install.sh` (Homebrew+Brewfile only), `restore.sh` (config only), `backup.sh` (reverse direction: live config → repo, then commit/push only if `git status --porcelain` is non-empty), `update.sh`, `check.sh`, `doctor.sh`, `services.sh`, `validate.sh`, `cleanup.sh`, `report.sh` (writes `reports/system-report.txt`), `inventory.sh` (writes 9 Markdown files under `reports/`: system/hardware/software/brew/fonts/extensions/services/databases/network), `preferences.sh` (`backup|restore|status` for macOS `defaults` domains, stored under `preferences/`), `release.sh` (semver bump + CHANGELOG draft + commit + tag + push — deliberately does *not* call `gh release create` itself since `release.yml` already does that when the tag lands). None of them duplicate logic that belongs in `common.sh`.
- **`devforgekit`** (repo root, no extension) — CLI dispatcher: parses `$1` as a command, shifts. `install`/`bootstrap` always `exec bootstrap.sh` directly (Node may not exist yet). Every other command: if `node` is on PATH and `cli/node_modules` exists, `exec node cli/bin/devforgekit.js "$cmd" "$@"`; otherwise falls back to the original `exec`-per-script case-arm table verbatim. Because it has no `.sh` extension, it's checked explicitly (not via `*.sh` globs) in `scripts/validate.sh` and `shellcheck.yml`/`lint.yml` — remember to add new bash scripts there too if they're ever extensionless (new commands implemented in `cli/` don't need this — they're plain `.js` files, not globbed by those bash-only checks).
- **`cli/`** (Layer 2, Node ESM) — the Core CLI: `src/index.js` builds one `commander` Program and registers one module per command from `src/commands/`; `src/core/*.js` holds the shared services (`paths`, `logger`, `errors`, `config`, `shell` — the spawn bridge back into `scripts/*.sh`, `registry`, `plugins`, `installer`, `health`). Commands that are pure bash wrappers (`update`, `backup`, `restore`, `check`, `doctor`, `validate`, `inventory`, `report`, `services`, `clean`, `release`, `preferences`) use `defineScriptCommand`/`runScript` in `core/shell.js` rather than reimplementing logic; `profile.js` is a hybrid (its `list`/`show`/`use` subcommands forward to `scripts/profile.sh` for the pre-existing Layer-1 Brewfile-subset picker, while `install`/`create`/`export`/`import`/`search`/`publish` are native, registry-driven); `config`, `component`, `search`, `collection`, `registry` (`generate`/`stats`), `stats`, `info`, `plugin`, and `recipe` (v1.2.1) are fully native. `installer.js`'s `resolveInstallOrder`/`installPlan` do dependency-graph resolution (topological sort, cycle detection); `cli/src/lib/installRunner.js` is the shared UI wrapper (`component install`, `collection install`, and `profile`/`recipe` `install`/`import` all use it) that prints the resolved plan, drives a progress bar, and reports PASS/skip/FAIL. See `docs/PlatformArchitecture.md` sections 1–2, 5 and `cli/README.md`.
- **`registry/`** (Layer 4) — `categories/*.yaml` (35), `packages/*.yaml` (261 components), `collections/*.yaml` (17 curated flat bundles), `profiles/*.yaml` (50 richer bundles that compose collections + extra components + suggested config `settings`), and `recipes/*.yaml` (8 built-in recipes — see the Recipe Engine bullet below), validated against `schema/{package,category,collection,profile,recipe}.schema.json` (ajv) by `cli/src/core/registry.js`, which also cross-checks referential integrity (every package's `category`/`dependencies`/`conflicts`, every collection's `components`, and every profile's/recipe's `collections`/`components` must resolve to something real — `checkIntegrity()`/`loadRegistry()`; this exact check caught a real duplicate-alias bug, `gcc` claimed by both `c` and `cpp`, before it shipped). Every package also carries the **Package Quality System** fields (v1.1.3): `documentation`, `architectures` (`intel`/`apple-silicon`/`linux` — CPU arch, distinct from the OS-level `platforms`; 86% coverage as of v2.1.1, up from 0% — a real gap `registry audit` surfaced on first use), `stability` (`stable`/`beta`/`deprecated`), `lastVerified` (`YYYY-MM-DD`), `ciVerified` (true only for the 5 packages `registry-smoke.yml` actually installs live). Install size/time are deliberately **not** stored fields — both are computed live (`info`'s `du -sh`; `installPlan`'s measured elapsed time) rather than fabricated, same reasoning both times. `cli/src/core/quality.js`'s `scoreManifest(pkg)` computes the objective, per-component **Manifest Quality Score** (v2.1.1: 13 equal-weight checks grouped into a `breakdown` by category — Metadata/Documentation/Reliability/Discoverability/Compatibility/Platform Support — replacing the old 10-check version's inflated "install/verify/uninstall tested" trio, which were all literally the same `ciVerified` boolean counted three times; synchronous, no network) and `checkLiveReachability(pkg)` upgrades the homepage/repository checks with a real `fetch` when explicitly requested (`devforgekit info <name> --live` — never automatic). `registry stats`' `qualityScore` is the registry-wide average of `scoreManifest`. `devforgekit registry audit` (v2.1.1, `computeRegistryAudit` in `commands/registry.js`) is a fourth, deliberately-distinct registry subcommand — a static (no live installs) curated health scorecard with coverage percentages and data-driven recommendations, also available as a dashboard page (`tui/pages/RegistryPage.js`, shortcut `y`). Drives `devforgekit component list/info/install/validate/repair/update/uninstall`, `devforgekit search <term> [--category/--tag]`, `devforgekit info <name> [--live]` (human-readable + quality score, distinct from `component info`'s raw JSON), `devforgekit collection list/info/install`, and `devforgekit profile list/show/use/install/create/export/import/search/publish`. `registry/registry.json` and `docs/Registry.md` are **generated** by `devforgekit registry generate` — never hand-edit them; CI (`cli.yml`) fails if they drift from a fresh run. `devforgekit registry stats` reports duplicate aliases, orphaned manifests, metadata completeness, and `qualityScore`/`ciVerifiedCount`. See `docs/PlatformArchitecture.md` sections 3 and 25.
- **`plugins/`** (Layer 3) — one directory per plugin, each with a `plugin.yml` manifest (schema v2 — `commands`/`events`/`dependencies` arrays, validated against `cli/src/schemas/plugin.schema.json`, `engine` semver-checked against `VERSION`) and command/hook scripts, discovered from both the in-repo `plugins/` and user-installed `~/.devforgekit/plugins/` (multi-root, same pattern as user profiles). `plugins/hello-world` is the working example registering a live `hello` command. The v1.2.0 Plugin SDK (`cli/src/core/pluginSdk.js`) adds the full local lifecycle — `devforgekit plugin create/test/build/package/publish/install` — plus `core/events.js`'s `install.beforeInstall`/`afterInstall` lifecycle events any plugin can subscribe to via its manifest's `events` array, and `core/signing.js`'s real Ed25519 signing/trust (your own key auto-trusted, third-party keys via `plugin trust`). "Sandbox" here means a `timeoutMs` execution limit, not a security boundary; "publish" stages a signed archive to a local/shared directory, not a hosted marketplace — that remains design-only. See `docs/PlatformArchitecture.md` section 4.
- **Recipe Engine** (v1.2.1, `cli/src/core/recipes.js` + `cli/src/commands/recipe.js`) — a lighter-weight, opinionated sibling of an environment profile: `expandRecipe` is a named wrapper around `expandProfile` (identical `collections`/`components` resolution), so `recipe install <name>` reuses the same `runInstallPlan` profiles use, then layers two things profiles don't have — `configure` steps (`git`/`vscode`/`cursor`/`shell`/`mise`, each a thin `bash -c 'source scripts/common.sh && restore_x'` call into the *exact* Layer 1 function `scripts/restore.sh` already runs, via `runConfigureStep` — zero new Layer 1 code) and a `verify` pass (`verifyComponents` runs every resolved component's `validate` command and reports PASS/FAIL/skip, distinct from `installPlan`'s silent skip decision). 8 built-in recipes ship under `registry/recipes/` (`ai-engineer`, `flutter-developer`, `backend-developer`, `devops-engineer`, `cybersecurity-lab`, `game-developer`, `ml-engineer`, `embedded-engineer`); user-created ones (`recipe create`) live at `~/.config/devforgekit/recipes/`, same multi-root discovery pattern as profiles/plugins. `recipe publish` is a deliberate stub, same as `profile publish`. See `docs/PlatformArchitecture.md` section 5 and `docs/Recipes.md`.
- **Interactive Terminal Dashboard** (v1.2.3, `cli/src/tui/`) — `devforgekit` with no arguments (or `dashboard`/`ui`) opens a full-screen Ink TUI; any argument at all takes the classic commander path unchanged, and non-TTY/`TERM=dumb`/`DEVFORGEKIT_NO_TUI=1` fall back to `--help` (`isTuiCapable` in `tui/index.js`; the root `devforgekit` dispatcher's no-arg branch just execs the Node bin and lets it decide). Written with `React.createElement` (aliased `h`) **without JSX** — the CLI has no build step and the TUI must not introduce one. Structure: `tui/index.js` (launch + the suspend/resume loop that unmounts Ink, hands the real TTY to terminal-owning work — doctor.sh/update.sh/inventory.sh/scaffolders/plugin commands — then re-renders on the page the user left), `App.js` (layout + global key router; menu vs content focus, single-letter page shortcuts only fire in menu focus; shortcut matching uses the last char of a chunk since keys can coalesce under load), `store.js` (one context+reducer; `tuiTheme` persisted via the normal config system), `theme.js` (20+ built-in themes, each a 30-token color-role object, plus WCAG contrast checking and custom-theme loading from `~/.config/devforgekit/themes/*.yaml` — the default `dark` theme got a full palette redesign in v1.4.0, see `docs/TUI.md`'s "Semantic tokens" section for the one real Ink limitation this ran into: `<Box>` has no `backgroundColor` support, only `<Text>` does, so panel/header/sidebar background tokens exist in the schema but aren't painted anywhere), `data.js` (cached wrappers over the real services — **no business logic in the TUI, ever**; the ~250 install-state probes run in small batches with timer yields so stdin never starves — this was a real, PTY-reproduced bug), `components/ui.js` (SelectList/Panel/TextField/Spinner/detailWidth — layout is responsive down to 80x24), `pages/` (one file per page). Engine hooks added for it (all additive, no existing caller changed): `runShellCommand`/`install`/`update`/`uninstall`/`repair`/`installPlan`/`runConfigureStep` accept optional `onOutput(text, stream)` to pipe child output into a render-safe log pane instead of inheriting the TTY (which would corrupt Ink's rendering). Honest-scoping rules apply: no mouse, no fake rollback/marketplace/scheduler/enable-disable controls, no screen-reader mode (the classic CLI is the accessible path) — pages state these inline. Tests: `cli/test/tui.test.js` (ink-testing-library over the real registry); real-PTY exit/suspend verified manually. See `docs/TUI.md` and `docs/PlatformArchitecture.md` section 20.
- **Workspace Manager** (v1.2.4, `cli/src/core/workspace/`) — makes an isolated per-project environment a single switchable unit: `devforgekit workspace switch <name>` applies git identity (`git.js`, real `git config --global` writes), SSH host identities (`ssh.js`, idempotent `~/.ssh/config` Host blocks via `markerBlock.js` — the same delimited-block-rewrite pattern the PATH manager uses, ported to Node), environment variables + AES-256-GCM-encrypted secrets (`env.js`, key at `~/.config/devforgekit/workspace-secret.key` mode 0600 — machine-local encryption, explicitly not a multi-user vault), Docker/Kubernetes context (`docker.js`/`kubernetes.js` — only switches into a context that already exists locally, never creates one), cloud CLI context (`cloud.js` — `AWS_PROFILE`/`GOOGLE_CLOUD_PROJECT`+`gcloud config set project` for AWS/GCP, reference-only for Azure/Firebase/Supabase/Cloudflare/Vercel/Netlify since those CLIs have no equivalent named-profile concept), and shell aliases/functions/PATH (`shellIntegration.js` — a generated `workspace-shell.sh` sourced by a one-line idempotent `.zshrc`/`.bashrc` hook, `mise activate`/`direnv hook`-style) — orchestrated by `switcher.js`, the only module that imports every subsystem. `store.js` owns CRUD + the active-workspace pointer at `~/.config/devforgekit/workspaces/<name>/workspace.json` (`schema.js`, ajv-validated against `cli/src/schemas/workspace.schema.json`, `schemaVersion: 2` as of the Compatibility Engine's `compatibility` field below — the first real entry in what had been an empty `migrateWorkspace` skeleton, which still throws rather than guessing on a newer-than-supported document). `health.js` reuses `core/health.js`'s `scoreResults` for `workspace verify` (same PASS/WARNING/FAIL weighting `check.sh`/`doctor` use); `snapshot.js` captures point-in-time JSON snapshots with diff/compare, and `switcher.js`'s `rollbackToSnapshot` takes an automatic safety snapshot first, then restores (and re-applies live if the target is the active workspace); `bundle.js` exports/imports portable `.tar.gz` bundles (secrets and snapshot history excluded by design; import auto-repairs dangling profile/recipe/component/plugin/collection references via the same pass `workspace repair` uses standalone). Optional `profile`/`collections`/`recipes`/`components`/`plugins` references resolve through the exact same `core/registry.js` functions `profile install`/`recipe install` use. 33 subcommands under `cli/src/commands/workspace.js` (30 plus the 3 `workspace compatibility scan/repair/history` subcommands the Compatibility Engine added); a 14th dashboard page (`tui/pages/WorkspacePage.js`, shortcut `w`) covers the high-frequency actions (browse/create/switch/verify/snapshot/delete) with the rest staying CLI-only, same scoping precedent as the Recipes/Doctor/Plugins pages. `devforgekit new` (via `commands/new.js`'s `recordProjectHistory`) appends generated projects to the active workspace's `projectHistory`. SSH trust is trust-on-first-use, identical to running `ssh` by hand. See `docs/PlatformArchitecture.md` section 21 and `docs/WorkspaceManager.md`.
- **Compatibility Engine** (v1.2.5, `cli/src/core/compatibility/`) — validates whether installed tools actually work *together*, not just whether each is individually installed: `rules.js` loads/schema-validates `registry/compatibility/*.yaml` (one file per tool, mirroring `registry/packages/`, integrity-checked against real package names the same way `registry.js`'s `checkIntegrity` already does) and merges in plugin-contributed rules (an optional `rules` field in `plugin.yml` — `core/plugins.js`'s `validatePluginManifest` now accepts a rules-only manifest, no `commands`/`events` required). `versionMatch.js` matches exact/minimum/maximum/range/wildcard/pre-release version expressions via the already-a-dependency `semver` package; `versions.js` detects an installed version honestly (an optional `versionCommand` package field, falling back to parsing `validate`'s output, else "unknown" — never guessed). `graph.js` builds the dependency graph and detects missing/circular/duplicate-tool problems (a genuinely new capability past `installer.js`'s own cycle detection, since it also walks compatibility-rule `requires` edges, not just package `dependencies`). `engine.js`'s `scanCompatibility()` is the one function every integration point calls, producing a 5-tier score (Healthy/Warning/Critical/Unsupported — an extension of `core/health.js`'s PASS/WARNING/FAIL formula; a Critical/Unsupported finding always wins the verdict regardless of the numeric score). `explain.js` gives a per-component ✓/✗ requirement breakdown; `repair.js`'s `planRepair`/`executeRepairPlan` turn a scan into a concrete plan (reusing `installer.js`'s `install`/`update`/`uninstall`) that only ever removes a conflicting package after an explicit confirm — never automatically. `report.js` exports Markdown/HTML/JSON (`--format pdf` is honestly labeled PDF-*ready* Markdown, not a binary PDF); `ai.js`'s `getAIRecommendations` is real as of v1.3.0's AI Development Assistant below — delegates to `core/ai/`'s provider/prompt machinery (the "explain" prompt kind) and throws the same clear "not configured"/"no API key" errors every `ai` command does when nothing is wired up, never a fake response. `compatibility scan --ai` is its CLI surface. Two packages the shipped rules needed didn't exist yet and were added: `registry/packages/xcode.yaml` (via the real `xcodes` brew formula) and `versionCommand`/`recommendedAlternatives` as new optional `package.schema.json` fields. Drives `devforgekit compatibility scan/check/explain/repair/graph/update/export`, wired into `doctor` (`--skip-compatibility`), `recipe install`/`profile install` (a pre-install check + displayed score), the Project Generator (opt-in per stack via a new `compatibilityCheck` generator field, wired for flutter/react-native/expo), the Workspace Manager (`workspace compatibility scan/repair/history`, backed by `workspace.schema.json`'s `schemaVersion: 2` `compatibility` field above), a 15th dashboard page (`tui/pages/CompatibilityPage.js`, shortcut `m`), and `registry stats`' `compatibilityCoverage`. See `docs/CompatibilityEngine.md`, `docs/RuleSchema.md`, `docs/CompatibilityRules.md`, and `docs/RepairGuide.md`.
- **Startup animation** (v1.2.6, `cli/src/tui/startup/`) — a sub-second boot sequence (blank → particles → the fixed "DevForgeKit" ASCII logo draws itself, left-aligned → wordmark/tagline fade in → a real boot checklist → hand off) that plays once before the dashboard's first Ink frame, never on suspend/resume and never for classic commands. `startupAnimation.js` orchestrates the 7 steps and is the only export `tui/index.js` calls, before `App`'s `render()` ever mounts; `asciiLogo.js`/`particleRenderer.js`/`loadingRenderer.js`/`transition.js` are pure/isolated building blocks with zero knowledge of Ink, React, or dashboard pages (and vice versa — this is a visual layer in front of the dashboard, not part of it). The boot checklist's tasks (`Loading registry`/`plugins`/`profiles`/`recipes`/`compatibility engine`/workspace manager/`Preparing dashboard`) are the same cached calls the dashboard's own pages already make through `data.js`, so the animation doubles as a cache warm-up rather than adding net startup time, and a task checks off the instant its real call resolves — never a fake timer. `shouldShowAnimation()` is the one gate for every disable path (non-TTY, CI, `TERM=dumb`, `DEVFORGEKIT_NO_ANIMATION=1`, `startupAnimation: false`, `startupAnimationSpeed: off`); `startupAnimationSpeed: fast` (reduced motion) skips particles and draws the logo instantly. ## Editing conventions specific to this repo

- `Brewfile` entries are grouped by package manager prefix (`brew`, `cask`, `vscode`, `npm`); keep new entries under the matching prefix and preserve the one-line comment above each `brew`/`cask` entry.
- Runtime versions belong in `mise.toml`, not `Brewfile`/`.zshrc`, unless the tool isn't mise-managed.
- `vscode/settings.json` and `cursor/settings.json` contain a live `mssql` connection profile block; treat any credentials-shaped fields there as sensitive even when blank, and never add real secrets to these files (they're copied verbatim to `$HOME` and committed). Real secrets go in a local `.env` (gitignored) following `.env.example`.
- When adding a new script, source `scripts/common.sh` the same way every existing script does (`SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"` then `source "$SCRIPT_DIR/common.sh"`), use `run_step`/`run_step_optional`/`record_result` for anything worth reporting, and end with `if print_summary; then exit 0; else exit 1; fi` (never a bare `print_summary; exit $?` — see the `set -e`/`pipefail` section above).
- Adding a new mirrored config file or `defaults` domain: add one line to `config_file_pairs()` or `preference_domain_pairs()` in `common.sh` — every script that iterates it (`backup.sh`/`restore.sh`, or `preferences.sh`) picks it up automatically. Don't hardcode the path/domain a second time in the calling script.
- Adding a new project template: create `templates/<name>/` with `README.md`, `.gitignore`, `.editorconfig`, `LICENSE` (copy verbatim from an existing template), and an example that actually runs — not a placeholder. If it has a `package.json`/`Dockerfile`/`.tf` file, add a matching entry to `.github/dependabot.yml`.
- Adding a new install profile: create `profiles/<name>/Brewfile` + `README.md` (first line becomes the one-line description in `./devforgekit profile list`) — no code changes needed, `profile_brewfile_path()` picks up any name with a matching file.
- Adding a directory the PATH manager should know about: one `label|directory` line in `path_manager_known_dirs()` (`common.sh`) — `doctor.sh`/`doctor.sh --fix` pick it up automatically.
- Adding a new `./devforgekit` command backed by bash: one `case` arm in the `devforgekit` file's legacy-fallback table that `exec`s the relevant script, **plus** a matching `cli/src/commands/<name>.js` using `defineScriptCommand` (see `cli/README.md`) so it also works once the Node CLI takes over — never put actual logic in `devforgekit` itself.
- Adding a new native (non-bash-backed) CLI command: one file under `cli/src/commands/`, registered in `cli/src/index.js`.
- Adding a new registry component: one `registry/packages/<name>.yaml` matching `registry/schema/package.schema.json` — no code changes needed, `devforgekit component list`/`search` pick it up automatically. If it depends on another component, add its name to `dependencies` (resolved automatically by `installer.js`). Set `stability`/`lastVerified` (today's date) honestly, and check `devforgekit info <name>` afterward — never hand-set `ciVerified: true` unless you've also added it to `.github/workflows/registry-smoke.yml`'s live-tested allowlist; the Manifest Quality Score is only useful if it reflects reality. Run `devforgekit registry generate` afterward to refresh the generated `registry/registry.json`/`docs/Registry.md` — CI fails the build if you forget.
- Adding a new collection: one `registry/collections/<name>.yaml` matching `registry/schema/collection.schema.json` (a `components` array of real package names) — no code changes needed.
- Adding a new profile: one `registry/profiles/<name>.yaml` matching `registry/schema/profile.schema.json` (at least one of `collections`/`components`, plus optional `settings`) — prefer referencing an existing collection over duplicating its component list. No code changes needed.
- Adding a new recipe: one `registry/recipes/<name>.yaml` matching `registry/schema/recipe.schema.json` (at least one of `collections`/`components`, plus optional `icon`/`tags`/`configure`/`verify`/`settings`) — no code changes needed, `devforgekit registry generate` afterward refreshes `registry.json`/`docs/Registry.md` (CI fails the build if you forget). `configure` entries must be one of the fixed set `core/recipes.js`'s `CONFIGURE_ACTIONS` implements (`git`/`vscode`/`cursor`/`shell`/`mise`) — tool-specific setup belongs on the package's own `post_install`, not a new `configure` action.
- Adding a new plugin: run `devforgekit plugin create <name>` to scaffold `plugin.yml`/`commands/`/`hooks/`/`tests/` matching `cli/src/schemas/plugin.schema.json` (schema v2 — `commands`/`events`/`dependencies`) — no code changes needed, `core/plugins.js` discovers and validates it automatically from either `plugins/` or `~/.devforgekit/plugins/`. Use `plugin test`/`build`/`package`/`install` to exercise the full SDK lifecycle before shipping one.
- Adding a new workspace subsystem (unlike the registry-driven extension points above, this one *does* need code — there's no YAML manifest for it): give it its own `cli/src/core/workspace/<name>.js` with an `apply*`/`capture*` pair matching `git.js`'s shape, add its schema block to `cli/src/schemas/workspace.schema.json`, wire it into `switcher.js`'s `switchToWorkspace`/`rollbackToSnapshot` and `health.js`'s `verifyWorkspace`, and add its CLI surface in `commands/workspace.js` — nothing else needs to change. Add a `cli/test/workspace-<name>.test.js` (temp `$HOME`, no mocks, same pattern as the existing ones).
- Adding a new project generator stack: one file under `cli/src/generators/<stack>.js` exporting `id`/`label`/`description` plus at least one of `scaffold`/`generate` (see `core/projectGenerator.js` for the full contract), then one import + array entry in `cli/src/generators/index.js` — `commands/new.js` and `core/projectGenerator.js` are both stack-agnostic, nothing else changes. Reuse `cli/src/generators/shared.js` for anything stack-agnostic (MIT license, `.editorconfig`, `.vscode/settings.json`, the common README shape, the shared Node CI workflow) instead of duplicating it. If the stack scaffolds via an external CLI, declare `requiresTool: { command, hint }` so a missing prerequisite fails with an actionable message before anything shells out. Optionally declare `compatibilityCheck: [componentNames]` to run a pre-scaffold compatibility check (see below) — most generators don't need one. Declare `recommends: [...]` (real `registry/packages/` names only) so Stack Intelligence (v2.1.2, see below) has something honest to show — never invent a package name that isn't a real registry entry. Do **not** write your own LICENSE file or hardcode MIT text — `runProjectGenerator`'s universal license step (`options.license`, default `mit`) handles every stack already; a generator-written LICENSE would just block a user's `--license` choice from applying to that one stack.
