# v3.0.2 Platform Stabilization Program

Tracking document for the pre-v3.0.2 platform stabilization initiative.
No new features during this phase - systematic validation, root-cause
analysis, architectural fixes, regression tests, and documentation for
every officially supported platform. See `docs/PlatformSupport.md` for
the resulting compatibility matrix (the source of truth going forward)
and `docs/NpmGlobalInstallRootCause.md` for the investigation that
triggered this program.

## Scope decision (recorded, not assumed)

This program's actual execution environment is a single Apple Silicon
(arm64) macOS machine with Docker (Linux containers only, arm64-native
and amd64-via-emulation). It has no access to: Intel Mac hardware/VMs, a
real Windows/WSL2 machine, or npm/Homebrew publish credentials/GitHub
release-creation permissions.

Decision (confirmed with the project owner): scope every phase to what
can be verified for real in this environment; phases requiring
unavailable hardware are explicitly marked **NOT TESTED** in
`docs/PlatformSupport.md` rather than assumed to pass. Phases requiring
publish credentials (npm publish, Homebrew tap update, GitHub Release
creation) are prepared in full but require the project owner's explicit
go-ahead and access to execute.

## Phase 3 (+ part of Phase 5): Ubuntu certification, with a Debian cross-check - 2026-07-14

**Objective (refined per project owner's guidance):** not just "does
`npm pack` install in a Docker container" - certify Ubuntu as a
first-class supported platform by reproducing a real first-time user's
full lifecycle: fresh OS, install Node.js the way a real user would,
confirm nothing assumes Homebrew exists, install `devforgekit`, exercise
every documented command surface, uninstall/reinstall, compare source
vs. npm install parity, and specifically look for Ubuntu/`apt`-specific
assumptions by cross-checking one other distro (Debian) side by side.

**Environment:** Docker Desktop on this machine (arm64 host; images
pulled/built are native `linux/arm64`, since that's what's actually
achievable in this environment - no `--platform linux/amd64` emulation
run this phase, see "Not tested this phase" below). Fixtures persisted
to `docker/platform-certification/` (Dockerfiles + usage) for reuse in
Phases 4/5/8, not left as throwaway `/tmp` scripts.

Two fresh, from-scratch images: `ubuntu:22.04` and `debian:12`, neither
with Homebrew installed, both with Node.js 20 via NodeSource (the real
path a fresh Ubuntu/Debian user takes - `apt`'s own `nodejs` is
typically too old) and npm force-upgraded to 11.18.0 to match the
`allow-scripts` gate that triggered the original root-owned-install bug.
A non-root `testuser` with passwordless `sudo` in both, plus `zsh`
installed alongside the default `bash`, for shell-detection testing. The
real tarball under test was built fresh from this checkout's current
source (`npm pack`), not the stale published `3.0.1` - so this phase's
results reflect the code as it exists right now, including the fix
found and applied during this same phase (see below).

### What was run for real (Ubuntu 22.04, then cross-checked on Debian 12)

4. **Full command surface** - `profile list`, `component list/info`,

6. **Shell detection** - `devforgekit --version`/`doctor` produce

### Bug found and fixed this phase

**`bootstrap.sh`'s non-macOS rejection printed a misleading "Bootstrap
aborted unexpectedly" message.** Root cause: a generic `EXIT` trap
(added for genuinely unexpected failures) fires on any non-zero exit,
including this deliberate, by-design early exit for non-macOS - so a
first-time Linux user running `devforgekit install --dry-run` saw
"aborted unexpectedly" appended after the actual, correct explanation,
which reads as "something went wrong" rather than "this is expected."
Minimal fix in `bootstrap.sh`: `trap - EXIT` immediately before this
specific `exit 1`, so the generic trap never fires for this expected
path, plus an expanded, actionable message (mirroring wording already
used elsewhere in the codebase for the same limitation) pointing the
user at running `devforgekit <command>` directly instead. Verified live
on both Ubuntu and Debian, via both the npm-installed copy and source
install, before and after the fix - confirmed the misleading line is
gone and only the clear explanation remains. No regression test added
(the trap/message is a one-line, non-macOS-only code path with no
existing bash test harness to hook into cheaply; the live Ubuntu/Debian
verification above is the evidence this phase produced instead - see
`Bug fixing discipline` guidance on not over-engineering a single-line
fix's test coverage).

### Investigated and classified as WARNING, not a bug

- **`npm install` inside `cli/` on Node <22.12 prints `npm warn
  EBADENGINE` for `commander@15.0.0`** (declares `engines.node:
  >=22.12.0`). Root-caused: Commander 15 is ESM-only and that floor
  exists specifically for CommonJS `require(esm)` support - it does not
  reflect an actual functional incompatibility for an ESM-native
  consumer. `cli/package.json` declares `"type": "module"` and uses
  native `import`, not `require()`, so this doesn't constrain
  devforgekit's real runtime behavior. Confirmed empirically: every
  command exercised this phase ran correctly on Node 20.20.2 (Ubuntu and
  Debian) with zero functional errors. No code change made - this is a
  cosmetic install-time warning from an upstream dependency's
  conservative `engines` field, not a devforgekit defect.
- **`npm uninstall -g devforgekit` leaves `~/.cache/devforgekit/cli-fallback/`
  behind** (the root-owned-install fallback mirror from item 3 above).
  Confirmed harmless (a few MB of orphaned cache, silently ignored by a
  future fresh install/mirror-rebuild) and confirmed not cheaply fixable
  the "right" way: npm's `allow-scripts` gate that this whole mirror
  mechanism exists to work around would equally block a `preuninstall`
  cleanup hook, so a real fix would need a different mechanism entirely
  (e.g. a `devforgekit uninstall` command that also cleans its own
  cache) rather than a one-line patch - out of scope to build
  speculatively during a certification phase. Documented in
  `docs/PlatformSupport.md`'s Known Limitations rather than patched.

### Not tested this phase (recorded, not assumed)

- **Fedora and Arch** - blocked by a local Docker Desktop environment
  issue on this machine unrelated to devforgekit: `docker-credential-desktop`'s
  symlink under `/usr/local/bin` breaks on every Docker Desktop launch
  because the app is being macOS-App-Translocated on each run (confirmed
  live via `ps aux` showing every running Docker process executing from a
  fresh `/private/var/folders/.../AppTranslocation/<new-uuid>/...` path
  each time, while the symlink points at whatever UUID was current the
  *previous* time it was written) - a `sudo xattr -d com.apple.quarantine`
  attempt to fix the underlying translocation trigger failed with
  "Operation not permitted" and was not pursued further at the project
  owner's direction, since it's a local machine/Docker Desktop issue, not
  a devforgekit issue. Dockerfiles for both (`docker/platform-certification/Dockerfile.fedora`,
  `Dockerfile.arch`) are written and ready to run as soon as image pulls
  work again - this is purely an execution blocker, not a scoping gap.
  The `dnf`/`pacman` code paths in `cli/src/core/platform/linux.js`
  remain architecturally present but empirically unverified on a real
  dnf/pacman machine.
- **Linux Mint / Pop!_OS** - not attempted this phase; both are
  Ubuntu-based, and this phase's own scope decision was to use Ubuntu as
  the reference and specifically diff against a *non*-Ubuntu-family
  distro (Debian) to surface Ubuntu-specific assumptions, which is a more
  informative use of this phase's time than re-testing two more
  Ubuntu-derivatives that would be very likely to just replicate the
  Ubuntu result.
- **`--platform linux/amd64` emulation** - not run this phase; every
  image built/pulled this phase was native `linux/arm64` (this host's
  architecture). Genuinely different behavior under amd64 emulation
  (e.g. any `os_arch`-conditional logic) is not yet verified on Linux.
- **WSL2-specific concerns** (Windows PATH leakage, `/mnt/c` interop,
  `explorer.exe`/`clip.exe` interop) - out of scope for Phase 3 by
  design; deferred to Phase 4 on real Windows/WSL2 hardware per the
  project owner's own proposed roadmap.
- **`scripts/cleanup.sh`, `scripts/update.sh`, `preferences.sh
  backup/restore`, `release.sh`/publish** - same reasoning as Phase 1:
  these mutate real state and weren't applicable/re-tested here (Linux
  containers are throwaway, so the "don't mutate the maintainer's real
  machine" concern from Phase 1 doesn't apply the same way, but
  `release.sh`/publish remain explicitly out of scope for any phase
  short of 13 regardless of environment).
